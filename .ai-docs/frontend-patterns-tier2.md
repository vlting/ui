## Cross-Repo Patterns

### CR-1: Derived State Over Synchronized State

**Description:** If a value can be computed from existing props or state, compute it during render. Do not store it in separate state and synchronize via effects. Both React Core and TanStack Query enforce this: React via "You Might Not Need an Effect," TanStack via "don't copy query data into local state."

**Example Code:**
```jsx
// BAD — synchronized state via effect
const [items, setItems] = useState([]);
const [count, setCount] = useState(0);
useEffect(() => { setCount(items.length); }, [items]); // double render

// GOOD — derived during render
const [items, setItems] = useState([]);
const count = items.length; // always in sync, single render
```

**Why It Exists:** Redundant state creates sync bugs — you must update every derived copy when the source changes. Computing during render eliminates an entire class of bugs and avoids re-render cascades.

**Trade-offs:** Expensive computations block the main thread. Use `useMemo` for genuinely slow calculations (measure first — 1ms+ threshold).

**When Not To Use It:** When the derivation involves an async operation (that's an effect — the network is an external system). When computation is genuinely expensive and dependencies change rarely.

**Source:** [You Might Not Need an Effect](https://react.dev/learn/you-might-not-need-an-effect), [Choosing the State Structure](https://react.dev/learn/choosing-the-state-structure), TkDodo: [React Query as a State Manager](https://tkdodo.eu/blog/react-query-as-a-state-manager)

**Impact:** Cross-repo consensus. Directly relevant to design system components — a Button's disabled visual state should derive from props, not be stored separately.

### CR-2: Controlled/Uncontrolled Duality

**Description:** Design system components should support both controlled (`value` + `onChange`) and uncontrolled (`defaultValue`) modes. React's own `<input>` does this. TanStack Query's forms guidance preserves this duality by deriving form state from server state for untouched fields while letting users edit freely.

**Why It Exists:** Different consumers have different needs. Simple prototypes want uncontrolled (less boilerplate). Production apps with validation, undo, or cross-field dependencies need controlled. A component must never switch modes mid-lifecycle.

**Trade-offs:** Increases implementation complexity. `defaultValue` updates after mount are intentionally ignored. `undefined` cannot be a valid controlled value (it signals "uncontrolled").

**When Not To Use It:** Pure display components with no interactive state. Components where uncontrolled mode makes no sense.

**Source:** [React input docs](https://react.dev/reference/react-dom/components/input), [React Sharing State](https://react.dev/learn/sharing-state-between-components), Also consensus in Tier 1 (React Aria + Radix)

**Impact:** Very high. This is the fundamental API design decision for every interactive component across all major libraries.

## React Core Patterns

### RC-1: Component Purity and Idempotent Rendering

**Description:** React components must be pure functions of their inputs. Given the same inputs, always return the same JSX. No mutations of external variables, no side effects during render.

**Why It Exists:** React's concurrent renderer can pause, abort, or restart rendering at any time. Impure renders fire unpredictably, cause duplicate mutations, or produce inconsistent UI. StrictMode double-invokes renders to surface violations. The React Compiler assumes purity for automatic memoization.

**Trade-offs:** Forces all side effects into event handlers or useEffect. Requires discipline.

**When Not To Use It:** Never violate this. It is not optional. Even "harmless" mutations during render break under concurrent features.

**Source:** [Components and Hooks must be pure](https://react.dev/reference/rules/components-and-hooks-must-be-pure), [The Rules of React — Sebastian Markbage](https://gist.github.com/sebmarkbage/75f0838967cd003cd7f9ab938eb1958f)

**Impact:** Foundation-level. Every other pattern depends on purity.

---

### RC-2: Effects Are Escape Hatches, Not Lifecycle Methods

**Description:** `useEffect` exists solely to synchronize with external systems (DOM APIs, network, third-party widgets, timers). It is NOT "run code after render."

**Example Code:**
```jsx
// BAD — effect to derive state
useEffect(() => { setFullName(firstName + ' ' + lastName); }, [firstName, lastName]);

// GOOD — calculate during render
const fullName = firstName + ' ' + lastName;
```

**Why It Exists:** Pre-hooks React conflated "mounted" with "start fetching" with "subscribe." The React team separated: rendering computes UI, event handlers respond to actions, effects synchronize with external world. This separation makes concurrent rendering possible.

**Trade-offs:** Requires unlearning class lifecycle models. Some use cases (analytics on mount, focus management) feel awkward as effects.

**When Not To Use It:** Transforming data for rendering. Responding to user events. Computing derived values. Caching expensive calculations (`useMemo`). Resetting state on prop change (`key`).

**Source:** [You Might Not Need an Effect](https://react.dev/learn/you-might-not-need-an-effect), [A Complete Guide to useEffect](https://overreacted.io/a-complete-guide-to-useeffect/)

**Impact:** Extremely high. Single most common source of bugs in React applications.

---

### RC-3: Key-Driven State Reset

**Description:** Instead of using effects to reset state when a prop changes, give the component a different `key`. React destroys and recreates the subtree with fresh state.

**Example Code:**
```jsx
// BAD — effect-based reset (flash of stale state, extra render)
useEffect(() => { setComment(''); }, [userId]);

// GOOD — key-driven reset (component never sees stale state)
<ProfilePage key={userId} />
```

**Why It Exists:** Effect-based reset renders once with stale state before firing, causing visual flash. Key approach is declarative — React handles full teardown/recreation.

**Trade-offs:** Destroys entire subtree (DOM nodes, scroll positions, uncontrolled input state). More expensive than selective reset for large subtrees.

**When Not To Use It:** When only a subset of state should reset. When subtree is expensive to mount. When you need animated transitions between states.

**Source:** [Preserving and Resetting State](https://react.dev/learn/preserving-and-resetting-state)

**Impact:** Critical for design system modals, forms, tab panels where state must reset on identity change.

### RC-4: Composition Over Configuration

**Description:** Prefer compound components (multiple small components composed via children) over monolithic prop-configured components.

**Example Code:**
```jsx
// BAD — configuration object (prop explosion)
<DataTable columns={[...]} sortable filterable onRowClick={...} />

// GOOD — compound composition
<Table>
  <Table.Header>
    <Table.Column sortable>Name</Table.Column>
  </Table.Header>
  <Table.Body>{rows.map(...)}</Table.Body>
</Table>
```

**Why It Exists:** Config components become prop-explosion nightmares as requirements grow. Compound components leverage React's composition model. Each sub-component is independently testable, stylable, and replaceable.

**Trade-offs:** More JSX verbosity for simple cases. Requires context for parent-child communication. Consumers must understand valid composition patterns. TypeScript typing is more complex.

**When Not To Use It:** Simple components with 2-3 props. Components where internal structure is fixed (e.g., Avatar).

**Source:** [Reusing Logic with Custom Hooks](https://react.dev/learn/reusing-logic-with-custom-hooks), Dominant pattern in Radix, Chakra, Headless UI, React Aria

**Impact:** Very high. Cross-library consensus — also a Tier 1 pattern (CR-4).

### RC-5: Composition Before Memoization

**Description:** Before reaching for `memo`/`useMemo`/`useCallback`, restructure components to move state down or lift content up. Passing unchanged `children` avoids re-renders without any memoization API.

**Example Code:**
```jsx
// BEFORE — ExpensiveTree re-renders on every keystroke
function App() {
  const [color, setColor] = useState('red');
  return <div style={{ color }}>
    <input value={color} onChange={e => setColor(e.target.value)} />
    <ExpensiveTree />
  </div>;
}

// AFTER — lift content up, ExpensiveTree doesn't re-render
function App() {
  return <ColorPicker><ExpensiveTree /></ColorPicker>;
}
function ColorPicker({ children }) {
  const [color, setColor] = useState('red');
  return <div style={{ color }}>
    <input value={color} onChange={e => setColor(e.target.value)} />
    {children}
  </div>;
}
```

**Why It Exists:** Dan Abramov's "Before You memo()" — structural techniques improve code architecture (better separation of concerns) with performance as a side effect. Memoization adds cognitive overhead, dependency tracking bugs, and memory cost.

**Trade-offs:** Requires restructuring component boundaries. Not always possible when changing state and expensive subtree are deeply intertwined.

**When Not To Use It:** When state and expensive rendering are inseparable. When profiling proves memo is needed. When the React Compiler handles it automatically.

**Source:** [Before You memo()](https://overreacted.io/before-you-memo/)

**Impact:** High for design system internals. Keeps components lean, avoids memo sprawl.

### RC-6: Writing Resilient Components

**Description:** Four principles from Dan Abramov: (1) Don't stop the data flow — props change, handle it. (2) Always be ready to render — never assume render frequency. (3) No component is a singleton — design as if rendered twice. (4) Keep local state truly local.

**Why It Exists:** Design systems are consumed by many teams in many contexts. Components must be resilient to unexpected rendering — different frequencies, positions in tree, rapidly changing props. The "rendered twice" test catches accidental singletons.

**Trade-offs:** Requires more forethought during component design. Can conflict with animation optimizations where intentionally showing stale data is the goal.

**When Not To Use It:** Always applicable. The design-time checklist for every component in a design system.

**Source:** [Writing Resilient Components](https://overreacted.io/writing-resilient-components/), [What Are the React Team Principles?](https://overreacted.io/what-are-the-react-team-principles/)

**Impact:** Very high. Predicts whether a component will survive real-world usage.

### RC-7: Custom Hooks as Focused Abstractions

**Description:** Extract reusable stateful logic into custom hooks with narrow, declarative APIs. A good custom hook constrains what it does — its name describes a single capability. Don't expose raw effects to consumers.

**Why It Exists:** React team guidance: "A good custom Hook makes the calling code more declarative by constraining what it does." Critical for design systems — consumers should call `useTooltip()` or `useDialog()`, not manually manage focus traps and ARIA attributes.

**Trade-offs:** Over-abstracting into too many tiny hooks fragments logic. Under-abstracting produces "god hooks." Requires upfront design effort.

**When Not To Use It:** One-off logic. Simple state that doesn't warrant a hook.

**Source:** [Reusing Logic with Custom Hooks](https://react.dev/learn/reusing-logic-with-custom-hooks)

**Impact:** Very high. Primary API surface for headless component libraries.

## TanStack Query Patterns

### TQ-1: Server State vs Client State Separation

**Description:** "Server state" (remote, async, shared ownership) and "client state" (synchronous, locally owned) are fundamentally different. Your frontend doesn't *own* server data — it has a *borrowed* recent version.

**Why It Exists:** Traditional state managers (Redux, MobX) treated all state uniformly, leading to manual cache management, stale data bugs, and enormous boilerplate. React Query is not a data fetching library — it is an async state manager. Data fetching (the network call) is trivial; managing the async lifecycle is the hard part.

**Trade-offs:** Two state management paradigms in your app. Must consciously decide which bucket each piece of state belongs to.

**When Not To Use It:** Purely local/ephemeral state (modal open/closed, form draft, UI toggles) should remain in client state.

**Source:** [Thinking in React Query](https://tkdodo.eu/blog/thinking-in-react-query), [React Query as a State Manager](https://tkdodo.eu/blog/react-query-as-a-state-manager)

**Impact:** The most influential mental model shift from TanStack Query.

---

### TQ-2: Stale/Fresh Caching Model

**Description:** Two independent timers per cache entry: `staleTime` (how long data is "fresh" — no refetch even on mount/focus) and `gcTime` (how long inactive data stays in memory). Defaults: `staleTime = 0` (instantly stale), `gcTime = 5 min`.

**Why It Exists:** Default `staleTime: 0` is intentionally aggressive — guarantees freshness via background refetch. The rename from `cacheTime` to `gcTime` in v5: almost everyone misunderstood `cacheTime` as "how long data is cached" when it actually means "how long unused data persists before garbage collection."

**Trade-offs:** High `staleTime` = fewer requests but staler data. `staleTime: Infinity` turns React Query into a "fetch once" tool.

**When Not To Use It:** Real-time data (chat, stock tickers) should keep `staleTime: 0` or use websocket-driven invalidation.

**Source:** [staleTime vs cacheTime discussion](https://github.com/TanStack/query/discussions/1685), [cacheTime rename issue #4678](https://github.com/TanStack/query/issues/4678)

**Impact:** The v5 rename was a breaking change affecting every user. Understanding this model is essential.

---

### TQ-3: Query Key Factories + queryOptions API

**Description:** Structure keys generic-to-specific: `['todos'] → ['todos', 'list', { filter }] → ['todos', 'detail', id]`. Co-locate keys with query functions using `queryOptions()` (v5).

**Example Code:**
```ts
const todosQueries = {
  all:    () => queryOptions({ queryKey: ['todos'], queryFn: fetchAllTodos }),
  list:   (filter: string) => queryOptions({
    queryKey: ['todos', 'list', { filter }],
    queryFn: () => fetchTodoList(filter),
  }),
  detail: (id: number) => queryOptions({
    queryKey: ['todos', 'detail', id],
    queryFn: () => fetchTodo(id),
  }),
}

// Usage — type-safe everywhere
useQuery(todosQueries.detail(5))
queryClient.invalidateQueries({ queryKey: ['todos'] }) // invalidates ALL
```

**Why It Exists:** TkDodo identified separating `queryKey` from `queryFn` as a design mistake — the key defines dependencies of the function, so they belong together. Generic-to-specific hierarchy enables fuzzy invalidation.

**Trade-offs:** Upfront investment. Over-granular keys create invalidation complexity; too-flat keys prevent targeted invalidation.

**When Not To Use It:** Trivial one-off queries that are never invalidated or shared.

**Source:** [Effective React Query Keys](https://tkdodo.eu/blog/effective-react-query-keys), [The Query Options API](https://tkdodo.eu/blog/the-query-options-api)

**Impact:** Most adopted community pattern. Formalized as official API in v5.

### TQ-4: Race Condition Prevention via Request Identity

**Description:** Automatically tracks the "latest" request per query key. Results from older in-flight requests silently discarded. Passes `AbortSignal` to query function for actual network cancellation.

**Why It Exists:** The classic `useEffect` + `fetch` pattern: search "books" then "movies," if "books" response arrives last, UI shows wrong data. React Query makes this impossible. TkDodo: "data fetching is simple; async state management is not."

**Trade-offs:** Cancellation only works if `queryFn` consumes the `signal`. Without it, the request still completes but results are ignored. Unmounted components' requests are NOT cancelled by default (data goes to cache).

**When Not To Use It:** Long-running requests you want alive after unmount benefit from the default non-cancellation.

**Source:** [Why You Want React Query](https://tkdodo.eu/blog/why-you-want-react-query), [Query Cancellation docs](https://tanstack.com/query/v4/docs/framework/react/guides/query-cancellation)

**Impact:** Primary selling point. The race condition problem is one of the most cited React footguns.

---

### TQ-5: Prefer Invalidation Over Direct Cache Updates

**Description:** After mutations, prefer `invalidateQueries` over `setQueryData`. Invalidation says "refetch" — simple, correct. Direct updates duplicate server-side logic on the client.

**Why It Exists:** TkDodo: "Most of the time, invalidation should be preferred over direct updates." Direct cache updates require knowing exactly what the server will do. Background refetches override manual cache changes anyway.

**Trade-offs:** Invalidation triggers a network round-trip. Optimistic updates (toggle buttons, like/unlike) give instant feedback but add rollback complexity.

**When Not To Use It:** Interactions requiring instant visual feedback benefit from optimistic updates despite added complexity.

**Source:** [Mastering Mutations](https://tkdodo.eu/blog/mastering-mutations-in-react-query), [Concurrent Optimistic Updates](https://tkdodo.eu/blog/concurrent-optimistic-updates-in-react-query)

**Impact:** Core TanStack Query recommendation.

---

### TQ-6: Mutation Callback Separation (Logic vs UI)

**Description:** Put logic callbacks (invalidation, cache updates) on `useMutation` options. Put UI callbacks (toasts, redirects) on the `mutate()` call site. `useMutation` callbacks always fire; `mutate()` callbacks don't fire if the component unmounts.

**Example Code:**
```ts
// Custom hook — logic (always fires)
function useUpdateTodo() {
  return useMutation({
    mutationFn: updateTodo,
    onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos'] }),
  })
}

// Component — UI (only fires if mounted)
mutation.mutate(data, {
  onSuccess: () => navigate('/todos'),
  onError: () => toast.error('Failed'),
})
```

**Why It Exists:** If invalidation lives in `mutate()` and the user navigates away during the mutation, the cache never gets invalidated. Separation keeps critical logic safe.

**Trade-offs:** Requires discipline. Developers accustomed to one place need to learn the callback execution model.

**When Not To Use It:** Simple mutations where component lifecycle is guaranteed.

**Source:** [Mastering Mutations](https://tkdodo.eu/blog/mastering-mutations-in-react-query)

**Impact:** One of the most commonly cited TanStack Query "gotchas."

### TQ-7: Request Waterfall Prevention

**Description:** The biggest performance footgun in component-level data fetching. A child's query doesn't start until the parent's resolves. Fix: prefetch in route loaders, `useSuspenseQueries` (not multiple `useSuspenseQuery` calls).

**Example Code:**
```ts
// ANTI-PATTERN — serial waterfall with Suspense
const { data: todos } = useSuspenseQuery(todosQueries.lists())
const { data: user } = useSuspenseQuery(userQueries.me())
// Second query never starts until first resolves

// FIX — parallel
const [{ data: todos }, { data: user }] = useSuspenseQueries({
  queries: [todosQueries.lists(), userQueries.me()],
})

// BEST — prefetch in route loader (no waterfall at all)
loader: ({ context: { queryClient } }) => {
  queryClient.ensureQueryData(todosQueries.lists())
  queryClient.ensureQueryData(userQueries.me())
}
```

**Why It Exists:** Suspense changes execution — a suspended component stops executing, so subsequent hooks never fire. Safe with `useQuery` (all hooks run unconditionally) but creates waterfalls with Suspense.

**Trade-offs:** Route-level prefetching couples data requirements to routing (less colocation). Dynamic/conditional queries harder to parallelize.

**When Not To Use It:** Truly dependent queries (fetch user, then user's permissions) are inherently sequential.

**Source:** [Request Waterfalls docs](https://tanstack.com/query/v5/docs/framework/react/guides/request-waterfalls), [Prefetching docs](https://tanstack.com/query/v5/docs/react/guides/prefetching)

**Impact:** Official docs call this "the biggest performance footgun."

## Anti-Patterns (identified by both teams)

| Anti-Pattern | Source | Alternative |
|---|---|---|
| Copying query data into Redux/Context/useState | TkDodo, multiple posts | Use React Query as the cache; derive locally |
| Using `useEffect` to derive state | React docs | Compute during render |
| Using `useEffect` to respond to events | React docs | Put logic in event handlers |
| Copying props into state ("stopping data flow") | Dan Abramov | Consume props directly; use `key` to reset |
| Using `onSuccess`/`onError` on `useQuery` | Removed in TanStack v5 | Use `useEffect` with `data`/`error` deps, or react to state |
| Multiple `useSuspenseQuery` calls in same component | TanStack docs | `useSuspenseQueries` for parallel |
| Putting invalidation logic in `mutate()` callbacks | TkDodo | Put in `useMutation` options |
| Adding memo/useMemo/useCallback before restructuring | Dan Abramov | Move state down, lift content up first |
| Exposing raw setters from custom hooks | React docs | Constrain API to declarative operations |
