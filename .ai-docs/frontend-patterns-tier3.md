## Cross-Repo Patterns

### CR-1: Concurrent Rendering Awareness

**Description:** Both Next.js (App Router) and React Native (New Architecture) now rely on React 18's concurrent features. Components must be pure (no side effects during render), handle interrupted renders gracefully, and support `useTransition` for deferrable updates. Design system components are consumed in both environments.

**Why It Exists:** React's concurrent renderer can pause, abort, or restart rendering. Next.js uses this for streaming + Suspense. React Native uses it for interruptible renders during gestures. A design system component that mutates during render or assumes single-pass rendering will break in both environments.

**Trade-offs:** Requires stricter component discipline. Some animation patterns (imperative mutations) conflict with purity. StrictMode double-invocation catches violations early.

**When Not To Use It:** Always applicable. This is the new baseline for React components.

**Source:** Next.js App Router architecture, React Native New Architecture (Fabric), React 18 concurrent features

**Impact:** Foundation-level. Both platforms depend on this.

---

### CR-2: Memoization as Last Resort, Not Default

**Description:** Both React Core docs and React Native performance guides explicitly state: `memo`/`useMemo`/`useCallback` carry comparison overhead per render. Only use when there's a measured performance problem. Fix the re-render source first (state colocation, context splitting, composition patterns).

**Why It Exists:** React docs: "If there is no perceptible lag when your component re-renders, memo is unnecessary." Over-memoization adds latency from unnecessary comparisons and creates a false sense of optimization. Design system library authors must decide what to memoize on behalf of all consumers — blanket memoization wastes budget on cheap components.

**Trade-offs:** `memo` is "completely useless if props are always different." The comparison cost can exceed re-render cost for simple components.

**When Not To Use It:** Cargo-cult memoization. Simple leaf components. Components where props structurally change every render. Fix root causes instead.

**Source:** [react.dev/reference/react/memo](https://react.dev/reference/react/memo), [Before You memo()](https://overreacted.io/before-you-memo/)

**Impact:** Cross-ecosystem consensus. Also echoed in Tier 2 (RC-5).

## Next.js Patterns

### NX-1: Granular `'use client'` Boundary Placement

**Description:** Place `'use client'` at leaf-level component files that actually need interactivity, not at barrel/index files. The directive defines a boundary in the module graph — everything it imports joins the client graph.

**Example Code:**
```tsx
// Button.tsx — needs onClick, gets directive
'use client'
export function Button({ onClick, children }) { ... }

// Card.tsx — purely presentational, NO directive
export function Card({ children }) { return <div className="card">{children}</div> }

// index.ts — barrel, NO directive here
export { Button } from './Button'
export { Card } from './Card'
```

**Why It Exists:** When `'use client'` is placed too high, it infects every imported module — all become client components, bundled into client JS, requiring hydration. The single most common App Router anti-pattern. Vercel's agent-skills repo lists this as CRITICAL-tier.

**Trade-offs:** More files to manage. Must audit each component for server compatibility. But the payoff is smaller client bundles and faster page loads.

**When Not To Use It:** If your entire library is inherently interactive (rich text editor), a top-level directive is fine.

**Source:** [Next.js Docs: Server and Client Components](https://nextjs.org/docs/app/getting-started/server-and-client-components), Vercel agent-skills CRITICAL-tier rule

**Impact:** Extremely high. The single highest-impact decision for design system App Router compatibility.

---

### NX-2: Donut Component / Children Slot Composition

**Description:** A Client Component defines a "hole" (via `children`) that a Server Component fills. The Server Component renders first; its output is serialized and slotted into the client tree. Client Components cannot import Server Components — but Server Component parents can pass them as children.

**Example Code:**
```tsx
// ClientShell.tsx
'use client'
export function ClientShell({ children }) {
  const [open, setOpen] = useState(false)
  return <div onClick={() => setOpen(!open)}>{children}</div>
}

// page.tsx (Server Component)
import { ClientShell } from './ClientShell'
import { ServerContent } from './ServerContent' // stays on server
export default function Page() {
  return <ClientShell><ServerContent /></ClientShell>
}
```

**Why It Exists:** Foundation of the Provider pattern in App Router. Server-rendered children pass through Client Component wrappers without being forced into the client bundle.

**Trade-offs:** Props at the boundary must be serializable (no functions, classes, Dates). Can't use `cloneElement` to add props to server-rendered children.

**When Not To Use It:** When the child genuinely needs client-side interactivity and will be a Client Component anyway.

**Source:** [Donut Components](https://frontendatscale.com/blog/donut-components/), [Next.js Composition Patterns](https://nextjs.org/docs/app/building-your-application/rendering/composition-patterns)

**Impact:** Very high. Canonical pattern for context providers, layout shells, theme wrappers.

### NX-3: Package Exports with `react-server` Condition

**Description:** Use `package.json` `"exports"` with the `"react-server"` condition to ship different entry points for server vs client resolution. Defined in React RFC #227 by Sebastian Markbage.

**Example Code:**
```json
{
  "exports": {
    ".": {
      "react-server": "./dist/server/index.js",
      "default": "./dist/client/index.js"
    }
  }
}
```

**Why It Exists:** Libraries referencing `useContext` for theming force every component to be a client component. Conditional exports let libraries ship server-optimized code alongside full client code. The bundler forks module resolution per environment.

**Trade-offs:** Requires two build outputs. Adds build pipeline complexity. Two copies of some modules may exist in the bundle.

**When Not To Use It:** Entirely client-side libraries. Libraries already using per-component `'use client'` directives.

**Source:** [React RFC #227: Server Module Conventions](https://github.com/reactjs/rfcs/blob/main/text/0227-server-module-conventions.md)

**Impact:** High. The official React-level mechanism. Radix and other modern libraries are adopting this.

---

### NX-4: Barrel File Optimization

**Description:** Avoid barrel files or ensure compatibility with `optimizePackageImports`. Barrel files with 200+ re-exports cause 200-800ms import overhead. Next.js rewrites imports at compile time to bypass barrels.

**Why It Exists:** Vercel engineering found barrel files are the primary cause of bundle bloat in component libraries. Tree-shaking doesn't help when externalized (SSR). Benchmarks: 28% faster builds, 40% faster cold starts.

**Trade-offs:** `optimizePackageImports` depends on detectable barrel patterns. Direct imports require consumers to know internal paths. Set `"sideEffects": false` in package.json.

**When Not To Use It:** Libraries with <20 exports and no heavy dependencies.

**Source:** [How We Optimized Package Imports in Next.js](https://vercel.com/blog/how-we-optimized-package-imports-in-next-js)

**Impact:** Very high. Vercel pre-configures this for Radix, MUI, Headless UI, lodash, date-fns, and more.

---

### NX-5: Zero-Runtime CSS for Server Components

**Description:** Avoid runtime CSS-in-JS (styled-components v5, Emotion's `css` prop) in Server Components. Use CSS Modules, Tailwind, CSS custom properties for tokens, or build-time extraction (vanilla-extract, Pigment CSS).

**Why It Exists:** Runtime CSS-in-JS injects styles via JS at render time — fundamentally incompatible with Server Components (no browser runtime on server). CSS custom properties provide dynamic theming without any JS runtime.

**Trade-offs:** Loses colocation and type-safety of CSS-in-JS. CSS Modules require separate files.

**When Not To Use It:** Purely client-side libraries where runtime CSS-in-JS is already working.

**Source:** [CSS in React Server Components (Josh W. Comeau)](https://www.joshwcomeau.com/react/css-in-rsc/), [Next.js CSS-in-JS docs](https://nextjs.org/docs/app/guides/css-in-js)

**Impact:** High. Forcing function for the entire CSS-in-JS ecosystem.

---

### NX-6: Streaming with Suspense Boundaries

**Description:** Wrap async Server Components in `<Suspense>` with fallback UI. The server streams the shell immediately and progressively fills in dynamic content. Foundation of Partial Prerendering (PPR) in Next.js 15+.

**Why It Exists:** Without Suspense, a slow fetch blocks the entire page. With PPR, Suspense fallbacks are prerendered at build time, dynamic content streams at request time — all in one HTTP request.

**Trade-offs:** Careful Suspense boundary placement needed — too many creates visual jank. Skeleton/Loader components must be first-class design system primitives.

**When Not To Use It:** Fully static pages with all data available at build time.

**Source:** [Next.js Partial Prerendering](https://nextjs.org/docs/15/app/getting-started/partial-prerendering), [Vercel Blog: PPR](https://vercel.com/blog/partial-prerendering-with-next-js-creating-a-new-default-rendering-model)

**Impact:** Very high. PPR is positioned as the "new default rendering model." Design systems need first-class Skeleton/Loader components.

---

### NX-7: Serialization-Aware Props

**Description:** Props crossing the server/client boundary must be serializable. Shape data on the server into exactly what the client needs — no functions, class instances, Dates, or deeply nested objects.

**Why It Exists:** The RSC protocol serializes all props into a "Flight payload." Large/nested props bloat the payload, delay hydration, and lock the main thread during parsing.

**Trade-offs:** Requires explicit data-shaping step. Can feel verbose. May require DTO types.

**When Not To Use It:** When both sides are Client Components. When data is already flat and small.

**Source:** [Hidden Performance Costs of RSC](https://dev.to/rbobr/the-hidden-performance-costs-of-react-server-components-248f)

**Impact:** High. One of the most common production performance issues — the Flight payload is invisible until you profile.

### NX-8: Provider Wrapper Pattern for Context

**Description:** React Context doesn't work in Server Components. Export a thin `'use client'` Provider wrapper accepting `children`. For truly server-compatible theming, use CSS custom properties instead.

**Example Code:**
```tsx
// ThemeProvider.tsx
'use client'
export function ThemeProvider({ theme, children }) {
  return <ThemeContext.Provider value={theme}>{children}</ThemeContext.Provider>
}

// layout.tsx (Server Component)
export default function RootLayout({ children }) {
  return <ThemeProvider theme="dark">{children}</ThemeProvider>
}
```

**Why It Exists:** Sebastian Markbage (RFC #227): "In component libraries like react-bootstrap, even simple components reference useContext for global themes, meaning you would need to re-export every single component from a wrapper." Solution: render providers deep, prefer CSS custom properties for server-compatible theming.

**Trade-offs:** Adds a client boundary at root layout. CSS custom properties avoid this entirely but lose React's component model for theme management.

**When Not To Use It:** If your design system uses CSS custom properties for all theming (no JS-driven theme logic).

**Source:** [Next.js Composition Patterns](https://nextjs.org/docs/14/app/building-your-application/rendering/composition-patterns), [React RFC #227](https://github.com/reactjs/rfcs/blob/main/text/0227-server-module-conventions.md)

**Impact:** Very high. Every design system consumed by Next.js apps must solve this.

## React Native Patterns

### RN-1: Native-Driven Animation Serialization

**Description:** The Animated API serializes the entire animation graph and sends it to the native UI thread once. After that, all frame calculations happen natively with zero JS bridge involvement. Only non-layout properties supported (`transform`, `opacity`).

**Why It Exists:** JS thread blocking causes dropped frames. Declarative graph on native thread = 60fps regardless of JS load.

**Trade-offs:** Only `transform`/`opacity` work with `useNativeDriver: true` — not `width`, `height`, `margin`. Once a value uses native driver, ALL animations on that value must. `PanResponder` incompatible — use direct events instead.

**When Not To Use It:** Animating layout/flexbox properties. Use `LayoutAnimation` or Reanimated worklets instead.

**Source:** [Using Native Driver for Animated](https://reactnative.dev/blog/2017/02/14/using-native-driver-for-animated)

**Impact:** Foundational — every RN animation tutorial references it.

---

### RN-2: Worklet Thread Isolation (Reanimated)

**Description:** Reanimated worklets are JS functions compiled to run on the UI thread. They solve gesture-driven animations needing sub-16ms response that the JS thread cannot guarantee.

**Trade-offs:** ~100 simultaneously animated components on low-end Android, ~500 on iOS. Debug mode perf significantly worse than release. Worklets have restricted closure scope — only serializable values can be captured.

**When Not To Use It:** Simple opacity/scale transitions where `Animated` + `useNativeDriver` suffices.

**Source:** [Reanimated Performance Guide](https://docs.swmansion.com/react-native-reanimated/docs/guides/performance/)

**Impact:** De facto standard for production gesture/animation work.

---

### RN-3: FlatList Virtualization Tuning

**Description:** FlatList renders only items within a configurable window. Five key props control blank-space-during-scroll vs memory vs JS responsiveness trade-offs.

**Key Optimizations:**
- `getItemLayout` eliminates async measurement (biggest single optimization)
- Memoize list items with `React.memo`
- Stable `renderItem` reference via `useCallback`
- Consider FlashList v2 for large lists (cell recycling, no manual size estimates)

**Trade-offs:** `removeClippedSubviews` can cause missing content on iOS. `getItemLayout` requires uniform heights. Over-tuned `windowSize` causes visible blanks during fast scroll.

**When Not To Use It:** Lists under ~50 items where `ScrollView` suffices.

**Source:** [Optimizing FlatList Configuration](https://reactnative.dev/docs/optimizing-flatlist-configuration)

**Impact:** Core RN documentation. Every production app with lists must tune these.

---

### RN-4: Platform Inconsistency Abstraction

**Description:** A design system must abstract known iOS/Android rendering differences behind a unified API. The framework deliberately does not normalize these — each platform has different user expectations.

**Key Inconsistencies:**

| Area | iOS | Android |
|------|-----|---------|
| Shadows | `shadowColor/Offset/Opacity/Radius` | `elevation` only |
| Text padding | None by default | Auto font padding |
| Touch feedback | Opacity change | Ripple effect |
| Default fonts | San Francisco | Roboto |
| TextInput multiline | Top-aligned | Center-aligned |

**Why It Exists:** Core maintainer Matt Oakes: "The defaults should lean towards being consistent with other apps on the same platform."

**Trade-offs:** Testing burden doubles. Some "inconsistencies" are intentional platform conventions. Must choose: pixel-identical cross-platform vs. platform-native feel.

**When Not To Use It:** When intentionally wanting platform-native feel (Material on Android, Cupertino on iOS).

**Source:** [RN Discussion #359](https://github.com/react-native-community/discussions-and-proposals/discussions/359)

**Impact:** Central tension in cross-platform design system work.

---

### RN-5: Platform-Specific File Extensions vs Platform.select

**Description:** Two approaches for platform divergence: `Platform.select()` for inline differences (styles, props), `.ios.ts`/`.android.ts` file extensions for structural differences. Metro resolves automatically.

**Example Code:**
```tsx
// Inline — small differences
...Platform.select({
  ios: { shadowColor: '#000', shadowOffset: {width: 0, height: 2} },
  android: { elevation: 4 },
  default: { boxShadow: '0 2px 4px rgba(0,0,0,0.2)' }, // web
})

// File extensions — large structural differences
// Button.ios.tsx / Button.android.tsx / Button.native.tsx
import Button from './Button'; // Metro resolves
```

**Trade-offs:** File extensions split logic across files. `Platform.select` grows unwieldy for complex differences. Simple ternary on `Platform.OS` is clearer for one-line differences.

**Source:** [Platform Specific Code docs](https://reactnative.dev/docs/platform-specific-code)

**Impact:** Foundational. Every cross-platform design system must choose a strategy.

---

### RN-6: Synchronous Layout with New Architecture

**Description:** Fabric renderer enables synchronous layout measurement via `useLayoutEffect`. Old architecture's async `onLayout` + `measureInWindow` caused visual jumps for tooltips/popovers.

**Why It Exists:** Tooltips, popovers, overlays need to measure targets then position relative to them. Async bridge made this a two-frame operation with visible flicker. New Architecture (default since RN 0.76) eliminates this.

**Trade-offs:** Requires New Architecture. `useLayoutEffect` blocks painting — keep it fast. Libraries must support both architectures during transition.

**Source:** [React Native Architecture](https://reactnative.dev/architecture/landing-page)

**Impact:** Architectural shift. Every component positioning overlays benefits.

---

### RN-7: InteractionManager for Deferred Work

**Description:** `InteractionManager.runAfterInteractions()` schedules work after all active animations and touches complete. Explicitly prioritizes animation smoothness over task completion speed.

**Why It Exists:** 16ms frame budget means any JS work during animation can cause jank. Defer data fetches, analytics, computation until interaction settles.

**Trade-offs:** Tasks delayed indefinitely during continuous interaction (long scroll). No priority system. Being partially superseded by concurrent rendering.

**When Not To Use It:** User-blocking tasks (auth, critical errors). When using New Architecture concurrent features.

**Source:** [InteractionManager docs](https://reactnative.dev/docs/interactionmanager)

**Impact:** Widely used. Still relevant for non-React async work.

## Anti-Patterns

| Anti-Pattern | Platform | Alternative |
|---|---|---|
| `'use client'` on barrel files | Next.js | Per-component directives at leaf level |
| Runtime CSS-in-JS in Server Components | Next.js | CSS Modules, Tailwind, CSS custom properties |
| Importing Server Components from Client Components | Next.js | Pass as `children` (donut pattern) |
| Non-serializable props across RSC boundary | Next.js | Shape DTOs on server, serialize Dates as strings |
| Animating layout properties with `useNativeDriver` | React Native | Use Reanimated worklets or LayoutAnimation |
| Heavy work during animations | React Native | `InteractionManager.runAfterInteractions()` |
| Blanket `React.memo` on all components | Both | Measure first, fix re-render source, memo last |
| `Platform.OS` ternaries for structural differences | React Native | File extensions (`.ios.tsx`/`.android.tsx`) |
| Single barrel file for 200+ components | Next.js | `optimizePackageImports` or direct imports |
| Capturing large closures in Reanimated worklets | React Native | Extract only needed values before worklet |
