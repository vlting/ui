## Cross-Repo Patterns (highest value — consensus across both libraries)

### CR-1: Controlled/Uncontrolled Duality Hook ✅

**Description:** A single hook (`useControlledState` / `useControllableState`) that lets any component work in both controlled (`value` + `onChange`) and uncontrolled (`defaultValue`) modes. Checks `prop !== undefined` to determine mode. Warns in dev if component switches modes.

**Example Code:**
```tsx
function useControllableState<T>({ prop, defaultProp, onChange }) {
  const isControlled = prop !== undefined;
  const [internal, setInternal] = useState(defaultProp);
  const value = isControlled ? prop : internal;
  const setValue = useCallback((next) => {
    const v = typeof next === 'function' ? next(value) : next;
    if (!isControlled) setInternal(v);
    if (!Object.is(value, v)) onChange?.(v);
  }, [isControlled, prop, onChange]);
  return [value, setValue];
}
```

**Why It Exists:** Every stateful component needs this dual personality. Without it, consumers must choose at design time whether to manage state. React Aria uses `useInsertionEffect` to reset value refs early, avoiding stale closures. Radix keeps it internal to preserve iteration freedom.

**Trade-offs:** Cannot use `undefined` as a valid controlled value. `onChange` fires per-call, not batched. Forces re-render in controlled mode to ensure layout effects run.

**When Not To Use It:** When you know you'll only ever need one mode. When `undefined` is a valid state value.

**Source:** React Aria `useControlledState.ts`, Radix `use-controllable-state.tsx`, 270+ npm packages depend on Radix's version despite its internal status

**Impact:** Foundation of every stateful component in both libraries. Cross-repo consensus.

### CR-2: Focus Scope with Containment and Restoration ✅

**Description:** A component managing focus for descendants with three capabilities: `contain` (trap focus, e.g. modals), `restoreFocus` (return focus to trigger on unmount), `autoFocus` (focus first focusable child on mount). Maintains an internal tree understanding React portals.

**Why It Exists:** Devon Govett's React RFC (#109) identifies five problems: refs everywhere, global focus state conflicts, arrow key nav, containment, portals breaking tab order. Proposed for React core, never adopted. Both React Aria and Radix ship their own implementations. Radix's `FocusScope` + `DismissableLayer` coordinate focus trapping with dismiss detection.

**Trade-offs:** Must intercept Tab key globally — "may not cover all cases across all browsers." Restoration fails when trigger unmounts before modal closes. FocusScope and DismissableLayer have inherent friction; Radix uses `requestAnimationFrame` delays to prevent false-positive blur detection.

**When Not To Use It:** Simple non-modal UIs where native tab order suffices.

**Source:** React RFC #109 (Devon Govett), Radix PR #191 (Modular Lock), Issue #1296 (focus thrashing)

**Impact:** Every overlay, dialog, popover, menu in both libraries depends on this.

### CR-3: Overlay/Modal Layer Stack Management ✅

**Description:** Coordinated system managing four concerns simultaneously for modals: (1) focus trapping, (2) scroll locking, (3) AT hiding of background (`aria-hidden`), (4) dismiss on outside click/Escape. Both libraries use depth-counting so only the deepest layer responds to Escape. React Aria uses `useModal` + `aria-hidden`. Radix uses `DismissableLayer` with pointer-event isolation.

**Why It Exists:** Getting any one concern wrong breaks accessibility. Nested modals (dropdown inside dialog) require coordinated stack — pressing Escape closes only the innermost layer. Radix's pointer-events cascade: body gets `pointer-events: none`, each layer gets `pointer-events: auto`.

**Trade-offs:** Radix's context instances must be consistent across component versions (layering breaks with npm deduplication failures). `pointer-events: none` can leak if components unmount incorrectly. iOS Safari ignores `overflow: hidden` on body, requiring touch-event interception.

**When Not To Use It:** Non-modal UI. Non-overlapping components.

**Source:** Radix PR #191, PR #1401 (pointer-events fix), React Aria `useModalOverlay`, `usePreventScroll`

**Impact:** Infrastructure enabling every modal/popover/dropdown to nest correctly.

---

### CR-4: Context-Based Compound Component Composition ✅

**Description:** Each component exposes a React Context for parent-child communication. Parent patterns pass hook results to children via context. Both libraries support "escape hatches" — custom implementations that drop into the composition tree.

**Why It Exists:** React Aria's 2023 RFC: hooks are powerful but "steep learning curve, APIs complex for simple use cases." Context composition eliminates glue code. Radix's `createContextScope` prevents context collision when compound components nest inside other compound components (e.g. `AlertDialog` wrapping `Dialog`).

**Trade-offs:** Radix's scoping adds `__scope*` props to every component (implementation leak). React Aria's collection system uses a "fake DOM" requiring two render passes. Radix filed React RFC #23287 for native scoping — React closed it.

**When Not To Use It:** Simple components that never compose inside other compound components.

**Source:** React Aria 2023 RFC, Radix `createContextScope`, React RFC #23287

**Impact:** The architectural foundation of both libraries' component APIs.

## React Aria Patterns

### RA-1: Three-Layer Architecture (State / Behavior / Component) ✅

**Description:** Every component decomposed into: (1) React Stately — platform-agnostic state hooks, (2) React Aria — behavior hooks returning DOM props (ARIA attrs, event handlers), (3) Themed component — thin shell composing layers 1+2.

**Example Code:**
```tsx
let state = useListState(props);           // Layer 1: State
let { listBoxProps } = useListBox(props, state, ref); // Layer 2: Behavior
return <div {...listBoxProps}>{/* items */}</div>;     // Layer 3: Component
```

**Why It Exists:** Devon Govett's 2019 RFC: "Many companies implementing component libraries from scratch. Properly implementing a11y, i18n, keyboard interactions is very hard." Separation lets any design system reuse battle-tested logic. HOCs and render props rejected — hooks compose more elegantly.

**Trade-offs:** RFC openly states: "The new architecture is quite a bit more complex. Previously fairly obvious where to look when a bug came up. Now many more possible places to look."

**When Not To Use It:** Simple one-off components. Overhead of three packages per component only pays off at scale.

**Source:** [2019-v3-architecture.md RFC](https://github.com/adobe/react-spectrum/blob/main/rfcs/2019-v3-architecture.md) — Devon Govett

**Impact:** Foundational. The canonical "headless UI" architecture reference.

### RA-2: Press Events Over Click Events (usePress) 📦

**Description:** Replaces browser `click` with a normalized `onPress` abstraction unifying mouse, touch, keyboard, and screen reader interactions. The hook (~1100 lines) is a state machine tracking `isPressed`, `isOverTarget`, `activePointerId`, `pointerType`, `ignoreEmulatedMouseEvents`.

**Why It Exists:** Problems with `onClick`: CSS `:active` persists on touch drag-off, screen readers fire only `onClick` with no preceding pointer events, emulated mouse events on touch cause double-firing, keyboard activation differs for links (Enter) vs buttons (Enter + Space). iOS Safari delays click events by 32ms on non-form elements. macOS has a bug where keyup doesn't fire while Meta is held.

**Trade-offs:** ~1100 lines for what seems simple. Justified by breadth of edge cases, each with linked browser bugs.

**When Not To Use It:** Don't use `usePress` directly for button semantics — use `useButton` (which composes `usePress`). `usePress` is the low-level primitive for building other ARIA patterns.

**Source:** [Building a Button Part 1](https://react-spectrum.adobe.com/blog/building-a-button-part-1.html), [usePress source](https://github.com/adobe/react-spectrum/blob/main/packages/@react-aria/interactions/src/usePress.ts)

**Impact:** Core primitive used by useButton, useCheckbox, useRadio, useTab, and dozens of other hooks.

### RA-3: Interaction Modality Tracking (useFocusVisible) 📦

**Description:** Global singleton tracks current interaction modality (`keyboard`, `pointer`, `virtual`) via document-level listeners. `useFocusVisible` and `useFocusRing` consume this to determine whether focus rings display. Focus with no preceding input event → `virtual` modality (screen reader).

**Why It Exists:** CSS `:focus-visible` implemented inconsistently across browsers — "the spec does not say when it should apply, browsers will likely implement different heuristics." JS-based approach guarantees consistency.

**Trade-offs:** Global mutable singleton. Firefox fires extra focus events when clicking into iframes. Text inputs show focus rings only for Tab/Escape, not regular typing.

**When Not To Use It:** If browser support allows relying on `:focus-visible` and you accept inconsistencies.

**Source:** [useFocusVisible.ts](https://github.com/adobe/react-spectrum/blob/main/packages/@react-aria/interactions/src/useFocusVisible.ts), [Discussion #5318](https://github.com/adobe/react-spectrum/discussions/5318)

**Impact:** Every focusable component depends on this.

### RA-4: Hover Interaction Isolation (useHover) 📦

**Description:** Replaces CSS `:hover` with a JS-based solution that only fires hover events when the pointer genuinely supports hovering (mouse, not touch).

**Why It Exists:** Devon Govett (Issue #771): "`:hover` does not play nice with touch devices. Usually they show hover styles on touch, then remove when you touch again." PostCSS `@media (hover: hover)` failed on hybrid devices (Surface, iPad with trackpad) — "detects whether hover is supported, not whether it's currently in use."

**Trade-offs:** Requires JS for what CSS "should" handle. Every hover-capable component needs the hook.

**When Not To Use It:** Desktop-only products. If you're fine with sticky hover states on touch.

**Source:** [Issue #771](https://github.com/adobe/react-spectrum/issues/771), [Building a Button Part 2](https://react-spectrum.adobe.com/blog/building-a-button-part-2.html)

**Impact:** Applied to every component with hover visual feedback.

### RA-5: Virtual Click Detection for Screen Readers 📦

**Description:** Heuristic system (`isVirtualClick`, `isVirtualPointerEvent`) detecting when a "click" was triggered by assistive technology rather than physical input. Enables components to set `pointerType: 'virtual'` and tailor behavior (e.g., skip ripple animations).

**Why It Exists:** Screen readers fire click events that look like mouse events but aren't. Zero-size pointer = screen reader. Android TalkBack special case: width=1, height=1, pressure=0, detail=0, pointerType='mouse'. JAWS/NVDA with Firefox: empty pointerType + trusted.

**Trade-offs:** Relies on browser-specific quirks that can break. `mozInputSource` (Firefox) now deprecated. Safari always reports `pressure === 0`, preventing its use as sole heuristic.

**When Not To Use It:** If you don't need to differentiate between physical and AT-triggered activations.

**Source:** [isVirtualEvent.ts](https://github.com/adobe/react-spectrum/blob/main/packages/@react-aria/utils/src/isVirtualEvent.ts), [Issue #3945](https://github.com/adobe/react-spectrum/issues/3945)

**Impact:** Critical for proper screen reader support across the library.

### RA-6: mergeProps for Hook Composability ✅

**Description:** Utility that merges multiple props objects: event handlers chained (both fire), classNames concatenated, ids deduplicated, all other props last-value-wins.

**Why It Exists:** Composable hooks architecture means a single DOM element receives props from multiple hooks (useButton + useFocusRing + useHover). Without mergeProps, developers manually chain every handler.

**Trade-offs:** Implicit behavior — if one handler calls `stopPropagation`, the other still fires. "Last wins" for non-special props can silently override.

**When Not To Use It:** When you need conditional prop merging or merge order matters semantically.

**Source:** [mergeProps docs](https://react-spectrum.adobe.com/react-aria/mergeProps.html)

**Impact:** Used in virtually every component composing multiple hooks.

### RX-2: Presence — Animation-Aware Mount/Unmount via State Machine ✅

**Description:** Suspends DOM unmounting until CSS exit animations complete. 3-state machine: `mounted` → `unmountSuspended` → `unmounted`. Detects animations by comparing computed `animationName` before and after `present` changes.

**Example Code (state machine):**
```ts
const [state, send] = useStateMachine(initialState, {
  mounted:          { UNMOUNT: 'unmounted', ANIMATION_OUT: 'unmountSuspended' },
  unmountSuspended: { MOUNT: 'mounted', ANIMATION_END: 'unmounted' },
  unmounted:        { MOUNT: 'mounted' },
});
```

**Why It Exists:** CSS animations need the DOM node during exit. React removes nodes immediately. `animationstart` fires after `animation-delay` (too late). No `animationrun` event exists. React 18 concurrency fix: temporarily set `animationFillMode: 'forwards'` to hold last keyframe until unmount completes.

**Trade-offs:** CSS animations only — transitions need separate handling. `forceMount` prop needed for JS animation libraries. Reading computed styles forces synchronous layout.

**When Not To Use It:** When using JS animation libraries exclusively (use `forceMount`).

**Source:** [Presence source](https://github.com/radix-ui/primitives/blob/main/packages/react/presence/src/presence.tsx), [Animation Guide](https://www.radix-ui.com/primitives/docs/guides/animation)

**Impact:** Widely used, frequently the source of animation-related issues in Radix ecosystem.

### RX-4: Roving Focus Group — Single-Tab-Stop Navigation ✅

**Description:** Only one item in a group is in the tab order. Arrow keys move focus between items. Tab exits the group entirely. Implements WAI-ARIA roving tabindex pattern.

**Why It Exists:** Without it, a toolbar with 20 buttons needs 20 tab presses. WAI-ARIA mandates roving tabindex for composite widgets (RadioGroup, Toolbar, Menu, Tabs).

**Trade-offs:** Can conflict with FocusScope in nested scenarios (Issue #1296: "focus thrashing within Dialog"). The group must know about all its items via internal Collection primitive.

**When Not To Use It:** Simple lists or nav links where sequential tab order is expected. The NavigationMenu spec explicitly avoids roving focus for website navigation — it "confuses users familiar with standard web patterns."

**Source:** [Accessibility Guide](https://www.radix-ui.com/primitives/docs/overview/accessibility), [Issue #1296](https://github.com/radix-ui/primitives/issues/1296)

**Impact:** Critical for WAI-ARIA compliance in composite widgets.

---

### RX-5: Selective WAI-ARIA Deviation — NavigationMenu ✅

**Description:** NavigationMenu deliberately does NOT use the `menu` ARIA role despite being a "menu." Uses `navigation` landmark + standard links instead.

**Why It Exists:** ARIA `menu`/`menubar` roles imply OS-level menu semantics: composite focus management, first-character navigation, typeahead. These confuse website users. "NavigationMenu does not use the WAI-ARIA menu role because menu and menubars behave like native operating system menus with complex functionality...often considered unnecessary for website navigation."

**Trade-offs:** Less "technically compliant" with strict ARIA reading. Requires understanding when ARIA patterns help vs. harm. Components that DO need menu semantics (DropdownMenu, ContextMenu) implement the full pattern.

**When Not To Use It:** Application menus (file menus, right-click) where OS-level semantics ARE expected.

**Source:** [NavigationMenu docs](https://www.radix-ui.com/primitives/docs/components/navigation-menu), [Accessibility Guide](https://www.radix-ui.com/primitives/docs/overview/accessibility)

**Impact:** Frequently cited as an example of pragmatic a11y thinking vs. dogmatic spec compliance.

## Anti-Patterns (identified by both teams)

| Anti-Pattern | Problem | Alternative |
|---|---|---|
| CSS `:hover` directly | Breaks on touch + hybrid devices | JS-based `useHover` with pointer type detection |
| CSS `:active` for press states | Persists on touch drag-off | `usePress` state machine |
| `onClick` directly | Doesn't normalize keyboard, touch, screen reader activation | `usePress` / `useButton` |
| CSS `:focus-visible` alone | Inconsistent browser heuristics | `useFocusVisible` / `useFocusRing` |
| Positive `tabIndex` values | Breaks expected tab order, a11y problems | Roving tabindex pattern |
| Sentinel DOM nodes for focus trap | Breaks CSS selectors and layout | Tab key interception |
| Switching controlled ↔ uncontrolled | Undefined behavior, data loss | Pick one mode; dev warning if switched |
