# Dynamic Component Route + Playground Component

Build the dynamic `/docs/components/[name]` route and the interactive playground component.

## Context
- Tech Spec: `.ai-epics/docs/docs-site/tech-spec.md`
- Component registry will be at `examples/docs/lib/registry.ts` (built by segment 001)
- API mapping loader will be at `examples/docs/lib/api-mapping.ts` (built by segment 001)
- CodeBlock, PropTable, ComponentPreview will be at `examples/docs/components/` (built by segment 002)
- Brand switching is handled by the Provider in the layout tree

## Instructions

### 1. Create `examples/docs/app/docs/components/[name]/page.tsx`

Dynamic server component route that renders a component documentation page.

```typescript
interface PageProps {
  params: Promise<{ name: string }>
}
```

Implementation:
- Use `getComponent(slug)` from the registry to look up the component
- If not found, call `notFound()` from `next/navigation`
- Load the api-mapping data using `loadApiMapping(slug)`
- Generate static params using `generateStaticParams()` that returns all component slugs from `getAllComponents()`
- Generate metadata using `generateMetadata()` for proper page titles

Page layout (top to bottom):
1. **Header**: Component name (h1), description, import path in a copyable code block
2. **Examples section**: For each example in the registry entry, render a `<ComponentPreview>` with the code. For the 10 exemplar components, render actual live components. For non-exemplar components, show a placeholder with just the code example.
3. **Playground section** (if playground config exists): Render the `<Playground>` component
4. **API Reference section**: Render `<PropTable>` from the api-mapping data. If compound component, render a table per sub-component.
5. **Notes section**: If the api-mapping has `notes`, render them as a bulleted list
6. **Migration section**: If the api-mapping has `breaking` differences, render a comparison table (shadcn → vlting)

For the live component rendering in examples:
- Import the actual components from `@vlting/ui` dynamically
- Use a `ComponentRenderer` client component that lazily loads components by name
- For the initial 10 exemplar components, hardcode the imports and example renderings in a lookup map

### 2. Create `examples/docs/components/component-renderer.tsx`

Client component that renders live component examples by name.

```typescript
interface ComponentRendererProps {
  componentName: string
  exampleName: string
}
```

This component uses a lookup map to render the correct example for a given component + example name combination. Initially includes renderings for the 10 exemplar components:

- **Button**: Basic button with text, variant showcase (all 7 variants), size showcase, loading state, with icon
- **Input**: Basic input, with placeholder, with label (using Field), with error state, disabled
- **Card**: Basic card with header/content/footer, elevated variant, interactive variant
- **Dialog**: Basic dialog with trigger, with form content, controlled dialog
- **Tabs**: Basic tabs with 3 panels, vertical tabs
- **Select**: Basic select with options, with groups, with placeholder
- **Accordion**: Single accordion, multiple accordion, collapsible
- **Table**: Basic table with header/body/footer, with caption
- **Form**: Basic form with validation fields
- **Tooltip**: Basic tooltip, different sides

Each example should be a self-contained React component that renders the `@vlting/ui` component with appropriate props and children.

### 3. Create `examples/docs/components/playground.tsx`

Interactive playground with prop controls for adjusting component props in real-time.

```typescript
interface PlaygroundConfig {
  component: string                    // Component name for dynamic rendering
  controls: PlaygroundControl[]        // List of adjustable props
}

interface PlaygroundControl {
  prop: string                        // Prop name
  type: 'select' | 'boolean' | 'string' | 'number'
  label: string                       // Display label
  options?: string[]                  // For 'select' type
  defaultValue: string | boolean | number
}
```

Implementation:
- Client component
- Render prop controls on the left/top (select dropdowns for enum props, checkboxes for booleans, text inputs for strings, number inputs for numbers)
- Render the live component preview on the right/bottom
- When a control changes, update the component's props in real-time
- Show the generated code below the preview (updating as props change)
- Use Tailwind for the controls UI (native HTML form elements — this is docs-site-only)
- Include playground configs for the 10 exemplar components (Button, Input, Card, Dialog, Tabs, Select, Accordion, Table, Form, Tooltip) in a `playgroundConfigs` map
