# Segment 1: NavigationMenu Home/End Keyboard Fix

Fix the NavigationMenu `Content` component to support Home/End keys for jumping to the first/last link. The `List` component already has this support — Content is the only gap.

## Scope

- `packages/components/NavigationMenu/NavigationMenu.tsx`
- `packages/components/NavigationMenu/NavigationMenu.test.tsx`
- `packages/components/NavigationMenu/NavigationMenu.spec.md`

## Instructions

### 1. Fix NavigationMenu.tsx

Open `packages/components/NavigationMenu/NavigationMenu.tsx`. Find the `Content` component's `handleKeyDown` callback (around line 211-239). It currently handles ArrowDown, ArrowUp, Escape, and Tab but is missing Home/End.

Add Home/End key handling **after** the ArrowUp case and **before** the Escape/Tab case:

```typescript
} else if (e.key === 'Home') {
  e.preventDefault()
  links[0]?.focus()
} else if (e.key === 'End') {
  e.preventDefault()
  links[links.length - 1]?.focus()
}
```

The `List` component (around line 138-145) already has Home/End as a reference — the Content implementation should mirror it but operate on `links` (anchor elements) instead of `triggers` (buttons).

### 2. Update NavigationMenu.spec.md

Open `packages/components/NavigationMenu/NavigationMenu.spec.md`. In the keyboard interaction section (Section 4), add explicit mention of Home/End support for Content:

Under the Content keyboard section, add:
- **Home**: Moves focus to the first link in the content panel
- **End**: Moves focus to the last link in the content panel

### 3. Update NavigationMenu.test.tsx

Open `packages/components/NavigationMenu/NavigationMenu.test.tsx`. Add test cases for Home/End key support on Content links.

Use the existing test patterns — import `render`, `screen`, `fireEvent` from `src/__test-utils__/render`. The test should:

1. Render a NavigationMenu with Content containing multiple links
2. Open the content panel
3. Fire Home key → verify first link gets focus
4. Fire End key → verify last link gets focus

Note: Some ARIA-related tests are skipped in this project due to Tamagui v2 RC JSDOM limitations. If the keyboard tests require the content to be visible (which depends on state), you may need to render with an initially open item. Check how existing tests handle this.
