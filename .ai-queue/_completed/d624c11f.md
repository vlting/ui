# Task: Add StlProvider to docs root providers + connect brand context

## Context
The docs app (`apps/docs/`) uses Next.js 15 + next-themes for dark mode + a custom `BrandProvider` for brand switching. However, `@vlting/ui` components require `StlProvider` (from `@vlting/stl-react`) for proper styling. Currently, no `StlProvider` exists in the provider chain, so ALL component previews render unstyled.

## Files to modify
- `apps/docs/components/providers.tsx` — ONLY this file

## Implementation

1. Import `StlProvider` from the stl-react package:
   ```ts
   import { StlProvider } from '../../../packages/stl-react/src/providers/StlProvider'
   ```

2. Import `injectBrandVars` from design-tokens:
   ```ts
   import { injectBrandVars } from '../../../packages/design-tokens'
   ```

3. Update the `BrandWrapper` component to also wrap children in `StlProvider`:
   - Read `resolvedTheme` from next-themes (already done)
   - Map `resolvedTheme` to StlProvider's `defaultColorMode` ('light' | 'dark')
   - Wrap children: `<StlProvider defaultColorMode={colorMode}>{children}</StlProvider>`

4. In the `useBrandCSSProperties` hook, also call `injectBrandVars(brandMap[brand], isDark ? 'dark' : 'light')` and apply the resulting CSS variables to the document root. This ensures STL components can access brand tokens.

5. The provider nesting order should be:
   ```
   ThemeProvider (next-themes)
     └─ BrandProvider (brand context)
         └─ BrandWrapper
             ├─ useBrandCSSProperties (CSS vars for Tailwind)
             ├─ injectBrandVars (CSS vars for STL)
             └─ StlProvider (defaultColorMode synced with next-themes)
                 └─ {children}
   ```

## Key Notes
- `StlProvider` manages its own color mode state. Pass `defaultColorMode` based on `resolvedTheme` from next-themes.
- The `fontConfig` property access (`definition.fontConfig`) may not exist on the `Brand` type — it should be `definition.fonts`. Check and fix if needed.
- Don't break existing functionality (brand switching, theme toggle).
