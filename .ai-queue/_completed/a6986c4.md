# Migrate Existing Component Specs to Unified Format

## Objective

Merge each component's `*.spec.md` + `*.contract.md` into a single unified `*.spec.md` following the new template from task 007. Delete the now-redundant `*.contract.md` files.

## Components to Migrate (5)

These components have both spec and contract files:
1. **Button** — `packages/components/Button/Button.spec.md` + `Button.contract.md`
2. **Card** — `packages/components/Card/Card.spec.md` + `Card.contract.md`
3. **Dialog** — `packages/components/Dialog/Dialog.spec.md` + `Dialog.contract.md`
4. **Input** — `packages/components/Input/Input.spec.md` + `Input.contract.md`
5. **Tabs** — `packages/components/Tabs/Tabs.spec.md` + `Tabs.contract.md`

## Migration Procedure

For each component:

1. **Read** the new template at `packages/_templates/component.spec.template.md`
2. **Read** the component's existing `*.spec.md` and `*.contract.md`
3. **Read** the component's TypeScript source (e.g., `Button.tsx`) to understand the actual typed API
4. **Create** a new unified `*.spec.md` that:
   - Follows the template structure exactly
   - Preserves all UX Intent content from the old spec (UX law references, mental models)
   - Preserves Behavioral Guarantees from the old contract
   - Preserves Breaking Change Criteria from the old contract
   - Replaces prop tables with a reference to TypeScript: "See `ButtonProps` in `Button.tsx`"
   - Adds the QUALITY_BASELINE.md reference at the top
   - Preserves Test Requirements from the old spec
   - Includes an Anatomy section describing sub-components without duplicating types
5. **Delete** the `*.contract.md` file (it's fully absorbed into the spec)

## Key Principle: No Type Duplication

The unified spec must NOT contain:
- Prop tables with TypeScript type columns
- Default value tables that mirror code
- Required/optional markers that TypeScript already provides

Instead, the Anatomy section should say things like:
> **Button** accepts `variant`, `tone`, `size`, `loading`, and `disabled` props to control appearance and state. See `ButtonProps` in `Button.tsx` for the full typed API.

Behavioral nuances that TypeScript CAN'T express should remain in prose:
> When `loading` is true, children are replaced with a Spinner and the button becomes disabled. Both `aria-busy` and `aria-disabled` are set.
