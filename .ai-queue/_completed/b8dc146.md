# Component Registry + API Mapping Loader

Build the core component registry that drives all component pages, and the API mapping loader utility.

## Context
- Tech Spec: `.ai-epics/docs/docs-site/tech-spec.md`
- 52 api-mapping.json files exist at `packages/components/*/api-mapping.json`
- Navigation structure exists at `examples/docs/lib/navigation.ts`
- This is the foundation data layer that all component pages will consume

## Instructions

### 1. Create `examples/docs/lib/registry.ts`

Build a component registry with metadata for ALL components. The registry maps component slugs to their metadata.

```typescript
export interface ComponentExample {
  name: string
  code: string
  description?: string
}

export interface ComponentEntry {
  name: string           // Display name: "Button"
  slug: string           // URL slug: "button"
  category: 'primitives' | 'components' | 'blocks' | 'charts' | 'icons'
  description: string    // Short description
  importPath: string     // e.g. "import { Button } from '@vlting/ui'"
  examples: ComponentExample[]
  apiMappingPath?: string  // relative path to api-mapping.json
  whenToUse?: string[]
  whenNotToUse?: string[]
  a11yNotes?: string[]
}
```

Include entries for ALL 52 components from the navigation. Each entry needs:
- Accurate `name` and `slug` (matching the navigation hrefs)
- A brief `description` (1 sentence describing what the component does)
- The correct `importPath` from `@vlting/ui`
- `apiMappingPath` pointing to the api-mapping.json (e.g., `../../packages/components/Button/api-mapping.json`)
- At least one basic example for each of the 10 exemplar components (Button, Input, Card, Dialog, Tabs, Select, Accordion, Table, Form, Tooltip)
- Simpler entries (name, slug, description, importPath, apiMappingPath) for the remaining ~42 components

Export a `getComponent(slug: string): ComponentEntry | undefined` helper and a `getAllComponents(): ComponentEntry[]` helper.

### 2. Create `examples/docs/lib/api-mapping.ts`

Build a utility to load and parse api-mapping.json files:

```typescript
export interface PropDef {
  type: string
  values?: string[]
  required?: boolean
  default?: string
  description?: string
}

export interface ApiMapping {
  component: string
  vlting: {
    import: string
    props: Record<string, PropDef | Record<string, PropDef>>
  }
  notes: string[]
  breaking: Array<{ shadcn: string; vlting: string; reason: string }>
}
```

Provide a `loadApiMapping(componentSlug: string): Promise<ApiMapping | null>` function that reads the JSON file for a given component. Use `fs.readFile` for server-side rendering (this will be called in server components).

Also provide a `flattenProps(props: Record<string, PropDef | Record<string, PropDef>>): Map<string, Record<string, PropDef>>` helper that normalizes the two api-mapping formats:
- Flat format (Button): `{ variant: PropDef, size: PropDef }` → Map with single entry `"Button" → props`
- Compound format (Dialog): `{ "Dialog.Root": {...}, "Dialog.Content": {...} }` → Map with entries per sub-component
