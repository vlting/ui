## Instructions

For EACH component/primitive listed above, create an `api-mapping.json` file following this schema:

```json
{
  "component": "ComponentName",
  "shadcn": {
    "import": "import { Component } from '@/components/ui/component'",
    "props": {
      "propName": { "type": "string", "values": ["val1", "val2"] }
    }
  },
  "vlting": {
    "import": "import { Component } from '@vlting/ui'",
    "props": {
      "propName": { "type": "string", "values": ["val1", "val2"], "mapping": { "shadcnVal": "vltingVal" } }
    }
  },
  "notes": ["Behavioral differences or migration tips"],
  "breaking": [
    { "shadcn": "className='...'", "vlting": "Use Tamagui style props", "reason": "Cross-platform support" }
  ]
}
```

**Key rules:**
1. Read the actual TypeScript types from each component's source file to get accurate prop types.
2. For components without shadcn equivalents (e.g., Direction, Item, Empty, Loader, Field, etc.), set `"shadcn": null` and include a note like "No direct shadcn equivalent."
3. For primitives, note that shadcn doesn't export these separately — they come from Radix or are inline. Set shadcn to the Radix equivalent or null.
4. Common breaking differences for ALL components:
   - `className` → Tamagui style props (cross-platform)
   - `asChild` → composition pattern (not supported in most vlting components)
   - `ref` → forwarded on most components
5. Keep JSON valid and consistently formatted.
6. Focus on accuracy over completeness — if you're unsure about a shadcn prop, include it with a `"TODO"` note rather than guessing.
