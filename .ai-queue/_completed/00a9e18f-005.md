## Context
useListState manages highlight/selection state for list-like components. Has prop-getter `getItemProps(index)` but returns standalone `onKeyDown` handler (not in a getter). Missing ARIA roles. Reimplements keyboard logic that exists in useKeyboardNavigation.

**Current API (from useListState.ts):**
- Props: `{ items: T[], defaultHighlightIndex? = 0, onSelect?, loop? = false }`
- Returns: `{ highlightIndex, highlightedItem, setHighlightIndex, getItemProps, onKeyDown }`
- getItemProps(index) returns: `{ 'aria-selected', onMouseEnter, onClick }` — MISSING `role: 'option'`
- onKeyDown handles: ArrowUp/Down, Home/End, Enter
- No container getter — onKeyDown is loose

## Implementation

### 1. Spec (useListState.spec.md)
Follow reference format (sections 1/2/4/5/7/8/9). Include:
- Section 4: Behavior — highlight management, keyboard navigation (ArrowUp/Down, Home/End, Enter for selection), mouse hover highlights, click selects, loop option (default false)
- Section 5: Accessibility — WCAG 4.1.2: getListProps returns `role="listbox"`, getItemProps returns `role="option"` + `aria-selected`
- Section 7: Composition — DX Note: onKeyDown handler duplicates useKeyboardNavigation logic (tech debt). Note: "should compose useKeyboardNavigation in a future refactor." Used by Select, Combobox, Menu.
- Section 8: Breaking changes — getItemProps shape, getListProps shape (new), keyboard behavior

### 2. Implementation Changes (useListState.ts)

**Add `getListProps` container getter:**
```typescript
const getListProps = useCallback(
  () => ({
    role: 'listbox' as const,
    onKeyDown,
  }),
  [onKeyDown],
)
```

**Add `role: 'option'` to getItemProps return.**

**Update UseListStateReturn interface** to include `getListProps` and updated getItemProps type.

**Keep standalone `onKeyDown` for backwards compat** — consumers may use it directly. But document getListProps as the preferred API.

function ListFixture<T>({ items, ...props }: UseListStateProps<T>) {
