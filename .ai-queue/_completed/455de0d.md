# API parity: update existing component variants to match shadcn/ui

## Goal

Update our existing components so their API (variants, props, sub-components) matches shadcn/ui's conventions. This ensures developers familiar with shadcn can use our library with minimal learning curve.

## Changes by Component

### Button (`packages/components/Button/Button.tsx`)

Current variants: `variant` (solid/outline/ghost), `tone` (neutral/primary/success/warning/danger), `size` (sm/md/lg)

**Add/update to match shadcn:**
- Add `variant: 'secondary'` — uses `$secondary` background
- Add `variant: 'destructive'` — uses destructive/red theme
- Add `variant: 'link'` — renders as text link with underline on hover
- Rename `solid` → `default` (keep `solid` as alias for backwards compat)
- Add `size: 'xs'` and `size: 'icon'` (square button), `size: 'icon-xs'`, `size: 'icon-sm'`, `size: 'icon-lg'`
- Add `asChild` prop support (renders children as the button element via Slot pattern)
- Keep our `tone` prop as an extension beyond shadcn's API
- Keep our `loading` prop (shadcn doesn't have this but it's useful)

### Input (`packages/components/Input/Input.tsx`)

Current props: `size`, `placeholder`, `value`, `onChangeText`, `label`, `helperText`, `error`, `disabled`, `leadingSlot`, `trailingSlot`

**Add/update:**
- Support `type` prop ('text' | 'password' | 'email' | 'number' | 'file' | 'search' | etc.)
- Add `aria-invalid` support (shadcn uses this for error state styling)
- Ensure `ref` forwarding works properly
- Support raw usage without label/helper (just the input element)

### Select (`packages/components/Select/Select.tsx`)

## General Patterns

For ALL variant additions:
1. Add new variant values to the `styled()` variant definition
2. Update the TypeScript props interface
3. Test that the new variants render correctly
4. Maintain backwards compatibility (don't remove existing variants, add aliases if renaming)
